netty学习1.1-传统IO(BIO)

发表于 2018-01-15 更新于 2020-11-04
本文字数： 3.7k 阅读时长 ≈ 3 分钟

传统BIO

在cmd里可以使用telent 端口号 链接
使用send 发送字符串

传统的BIO一共有两个堵塞点

  • 等待客户端的链接Socket socket = serverSocket.accept();
  • 等待链接客户端的输入完成(就绪)int data = is.read(b);

特点即缺点

  • 一个服务端对应一个客户端

在这个例子里可以把 服务端 比喻成 服务员 ,把 客户端 比喻成 顾客 在这个例子就是一个 服务员 服务一个顾客.没有被服务的在门外排队

接下来将是这个模型的改进版本 多线程BIO

多线程BIO

这个版本改进了 传统BIO 应对多个客户端需要等待的问题.
在这个例子就是为每一个 顾客 都安排一个 服务员 来者不拒.
在实际情况下这个是 非常糟糕的 在高并发的情况下这回影响到系统资源. 没有一家店会拥有无限多的 服务员.

那么解决这种问题就要用到线程池

伪异步IO版本(线程池BIO)

在这个模型中使用了线程池,使用了缓存.最最重要的点就是复用同一个线程复用多次.
无论 顾客 有多少 服务员 只有100个,没有排到的就等待

BIO总结

伪异步BIO看起来已经是终极解决方案了,但是依然存在如下的问题

  • 每当一个新的客户端请求接入时，服务端必须创建一个线程来处理这条链路
  • 如果在高性能高并发场景下肯定是没法用的

总的来说在BIO模型中是一问一答

接下来只能抛弃传统的BIO,引入新的NIO(JKD1.4就已经存在)

参数 说明
port 端口号
logfile 指定了记录日志的文件
dir 数据目录,数据库的写入会在这个目录.rdb,aof文件也会写在这个目录.
cluster-enabled 是否开启集群
cluster-config-file nodes.conf 集群配置文件的名称,每个节点都有一个集群相关的配置文件,持久化保存集群的信息.这个文件并不需要手动配置,这个配置文件有Redis生成并更新,每个Redis集群节点需要一个单独的配置文件,请确保与实例运行的系统中配置文件名称不冲突
cluster-node-timeout 节点互连超时的阀值, 集群节点超时毫秒数
appendonly 默认redis使用的是rdb方式持久化,这种方式在许多应用中已经足够用了.但是redis如果中途宕机,会导致可能有几分钟的数据丢失,根据save来策略进行持久化,Append Only File是另一种持久化方式,可以提供更好的持久化特性.Redis会把每次写入的数据在接收后都写入 appendonly.aof 文件,每次启动时Redis都会先把这个文件的数据读入内存里,先忽略RDB文件.
